# 수납 관리 통합 설계 (창구수납 + 수납 내역)

- 날짜: 2026-06-15
- 상태: 승인됨
- 관련 화면: `/payments`(창구수납), `/payment-history`(수납 내역)

## 배경 / 문제

현재 수납 관련 화면이 둘로 나뉘어 있다.

- **창구수납** (`app/(admin)/payments/page.tsx`): 원생 중심 처리(write) 화면. 좌측 이름 검색 → 원생 선택 → 우측에 청구 세부내역·수납 이력·`수납 입력`/`문자 발송` 버튼.
- **수납 내역** (`app/(admin)/payment-history/page.tsx`): 월별 현황 조회(read) 화면. 요약 카드 3개 + 월/상태/수단 필터 + 전체 테이블.

두 화면은 같은 데이터(`invoices`/`payments`/`students`/`classes`)를 다른 각도로 보여줄 뿐이며, 창구수납의 "수납 이력" 탭은 수납 내역 테이블의 1인분 버전이다. 중복이 크고, "현황을 보다가 미납자를 바로 처리"하는 동선이 화면 전환으로 끊긴다.

## 목표

두 화면을 단일 **"수납 관리"** 화면으로 통합한다. 평소엔 전체 현황(overview), 행 클릭 한 번으로 그 자리에서 처리(detail)로 이어지는 **현황 우선(overview-first) 마스터-디테일** 구조.

## 결정 사항 (확정)

1. **기본 레이아웃**: 현황 우선 마스터-디테일. 선택 없을 때 전체 폭 테이블, 행 선택 시 목록 축소 + 우측 디테일 패널.
2. **라우팅**: `/payments` 하나로 통합. `/payment-history`는 `/payments`로 redirect. 사이드바 2개 항목 → "수납 관리" 1개.
3. **수납 입력 방식**: 기존 모달 팝업 유지 (인라인화하지 않음).

## 상세 설계

### 파일 변경

- `app/(admin)/payments/page.tsx` → "수납 관리"로 재작성 (메인 화면, 두 화면 로직 통합)
- `app/(admin)/payment-history/page.tsx` → 서버 컴포넌트 스텁으로 교체, `redirect('/payments')` 호출 (`next/navigation`)
- `components/layout/Sidebar.tsx` → `창구수납`/`수납 내역` 두 nav 항목을 `수납 관리`(`/payments`) 한 개로 통합. 아이콘은 기존 창구수납 아이콘 사용.

### 레이아웃

```
수납 관리
[총 수납액] [미납 합계] [전체 N명]            ← 요약 카드 3개 (월 필터 반영)
[2026-06▾] [상태▾] [수단▾] [🔍 이름검색]      ← 필터바 + 검색 통합

선택 없음 → 전체 폭 테이블 (현황 모드)
  원생 | 반 | 상태 | 수단 | 금액 | 수납일

행 선택 → 좌측 목록 축소 + 우측 디테일
  ┌ 목록(원생/상태/금액) ┬ 디테일 ──────────────┐
  │ ▶정도현 미납 21만     │ 정도현 요약 + 버튼     │
  │  김민준 완납 21만     │ 청구 세부내역          │
  │  …                    │ 수납 이력(이 학생)     │
  └───────────────────────┴───────────────────────┘
```

### 동작 규칙

- **요약 카드 + 필터**: 수납 내역의 집계/필터를 그대로 사용. 검색어(이름 부분일치)를 필터에 추가 → 월·상태·수단·이름 AND 결합.
- **현황 테이블**: 선택 전 전체 폭. 행 클릭 → 학생 선택, 테이블 컬럼 축소(원생/상태/금액)되며 디테일 등장.
- **디테일 패널**: 창구수납 우측(원생 요약 + `문자 발송`/`수납 입력` 버튼 + 청구 세부내역 + 수납 이력 탭)을 이식.
- **수납 입력 / 문자 발송**: 기존 모달 2개 그대로 재사용.
- **선택 해제**: 디테일 패널 닫기 → 전체 폭 현황으로 복귀.

### 데이터 / 로직

- 신규 데이터 모델 불필요. `invoices`/`payments`/`students`/`classes` 그대로.
- 수납 내역의 `rows` 집계 + 창구수납의 `studentInvoices`/`studentPayments`/`totalAmount` 계산을 한 컴포넌트로 통합.
- 선택 상태는 `selectedStudent`(또는 선택된 invoice row) 하나로 관리.

## 범위 밖 (YAGNI)

- 목록 행 인라인 빠른 버튼(문자/수납) — 디테일 경유로 충분.
- 수납 입력 폼 인라인화 — 모달 유지.
- 권한 분리(창구/관리자) — 현재 단일 staff 스코프라 불필요.

## 검증

- `/payments`에서: 진입 시 전체 현황 테이블 표시 → 월/상태/수단/이름 필터 동작 → 행 클릭 시 디테일 등장 → `수납 입력`/`문자 발송` 모달 동작 → 디테일 닫기 시 현황 복귀.
- `/payment-history` 접근 시 `/payments`로 redirect.
- 사이드바에 "수납 관리" 1개만 노출, 활성 상태 표시 정상.
