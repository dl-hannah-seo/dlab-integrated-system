# 시간표 — 정기 + 보강(변동) 일정 설계 스펙

**날짜:** 2026-06-16
**상태:** 초안 (리뷰 대기)
**관련 스펙:** [2026-06-16-timetable-design.md](2026-06-16-timetable-design.md) (정기 시간표 1차)
**대상 라우트:** `/schedule`

---

## 배경 및 목적

현재 `/schedule` 시간표는 **학기 내내 반복되는 정기 편성** 한 겹만 표현한다.
- `ClassGroup` = `day_group`('토','화목') + `time_slot`('0900') → **요일·시간 패턴**만 있고 **특정 날짜(달력) 개념이 없다.**
- 그래서 "이번 주 토요일 10시 보강", "다음 주 화요일 휴강" 같은 **주차별 변동을 전혀 표현할 수 없다.** (1차 스펙에서 `보강 수업 별도 표시`는 명시적으로 범위 외였다.)

실제 운영에서는 **매주 상당 부분이 달라진다**(보강·휴강·시간변경·특강). 따라서 시간표는 **고정 시간표(정기 편성) + 주차별 실제 일정(변동)** 두 층으로 나뉘어야 한다.

---

## 핵심 결정 — 모델 방향

세 가지 방향을 검토했다.

| 방향 | 설명 | 평가 |
|------|------|------|
| A. 정기 base + 날짜별 예외 레이어 | 정기는 그대로, 변경분만 예외로 얹음 | 변동이 가끔일 때 최적. **매주 상당 부분 변동 시 예외가 폭증**해 base가 무의미 |
| **B. 주차별 세션 전개 (+ 정기 템플릿 앵커) ⭐** | 정기 편성을 **템플릿**으로 두고 주차별 `Session`을 생성·개별 편집 | 매주 변동이 큰 현 운영에 적합. 정기는 **생성 소스 + 청구 기준**으로 유지 |
| C. 정기만 유지, 보강은 별도 | 시간표 모델 불변, 보강은 메모/출결에서만 | 시간표에 보강이 안 보임 — 요구 미충족 |

**채택: B (주차별 세션 전개 + 정기 템플릿 앵커).**
"매주 상당 부분 달라짐"이라는 운영 특성상 변경을 1급 데이터(`Session`)로 다루되, 수강 등록·수강료가 이미 `Class`(정기 편성)에 묶여 있으므로 **정기 편성을 버리지 않고 템플릿/청구 기준으로 남긴다.**

## 데이터 모델

### 유지 (정기 = 템플릿 · 청구 기준)
- `ClassGroup` — 요일·시간 패턴 (생성 규칙의 소스)
- `Class` — 수업명·담당·정원·기간·**수강료** (수강 등록과 청구의 기준, 변동과 무관)

### 신규 — `Session` (주차별 실제 수업 1회)

```ts
export interface Session {
  id: string;
  class_id: string;          // 파생 원본 정기 반 (Class.id)
  date: string;              // 'YYYY-MM-DD' — 특정 날짜 (달력 1급 개념)
  start_time: string;        // 'HHMM'
  end_time?: string;         // 'HHMM' (선택)
  type: '정규' | '보강' | '휴강' | '특강';
  teacher?: string;          // 변경 시 override (없으면 Class.teacher 사용)
  room?: string;             // 강의실 변경 표시용 (선택)
  replaces_session_id?: string; // 보강이 대체하는 휴강 세션 (선택)
  memo?: string;             // 사유/비고
}
```

**세션 유형 의미**
- `정규` — 정기 템플릿에서 자동 생성된 그대로의 세션
- `휴강` — 그 날짜의 정규 세션이 취소됨 (시간표에 취소선/흐림으로 잔존)
- `보강` — 휴강 등을 보충하는 추가 세션 (`replaces_session_id`로 연결 가능)
- `특강` — 정기와 무관한 일회성 수업

### 생성 규칙
선택한 주차(날짜 범위)에 대해, 각 `Class`의 `ClassGroup`(요일×시간)을 `[start_date, end_date]` 내에서 펼쳐 **`정규` 세션을 생성**한다. 그 위에서 사용자가 `휴강` 표시 / `보강`·`특강` 추가로 편집한다.
- 생성 시점은 **주 단위 lazy 생성**(보고 있는 주차만 펼침)을 기본으로 한다. → [열린 질문 1]

## 페이지 구조 변경

### 헤더 (추가)
- 기존: 제목 · 학기 선택 · 그리드/카드 토글
- **추가: 주차 선택기** `◀  2026-07-06 ~ 07-12  ▶` (기본값: 오늘이 속한 주)
- **추가: 보기 모드** — `이번 주 실제` ↔ `정기 편성`(템플릿, 날짜 무시 = 현재 동작) 토글

### 그리드 뷰
- 셀의 출처가 **정기 편성 → 그 주차의 `Session` 목록**으로 바뀐다. (레이아웃·09~18 축·한 칸 복수 누적은 그대로 재사용)
- **세션 유형별 표시**
  - `정규` — 현행 블록
  - `보강` — 강조색(예: 오렌지 계열) + `보강` 배지
  - `특강` — 별도 색 + `특강` 배지
  - `휴강` — 취소선 + 흐림(opacity↓), 클릭은 가능(사유 확인)
  - 시간/강사/강의실이 템플릿과 다르면 `시간변경` 등 배지

### 팝오버 (추가 필드)
- 기존(수업명·요일·담당·인원·기간·요금)에 더해: **세션 유형 · 날짜 · (보강이면) 대체 대상 · 메모**

## 다른 화면과의 연동 (경계)

- **수강 등록 / 수강료** — `Class`(정기) 기준 유지. **세션 변동은 청구에 영향 없음**(보강은 기본 무상 가정). → [열린 질문 2]
- **출결** — 출결은 본래 "그 날짜의 수업"이므로 장기적으로 **`Session` 단위 연결**이 자연스럽다. 이번 범위에서는 설계만 언급, 연동은 후속.
- **정원/인원** — 인원은 `Class` 기준. 보강에 일부만 참석하는 케이스는 이번 범위 밖. → [열린 질문 3]

## 열린 질문 (리뷰에서 확정)

1. **세션 생성 시점** — 주 단위 lazy 생성(권장) vs 학기 전체 미리 전개. 어느 쪽?
2. **보강 비용** — 항상 무상인가, 일부 유상 케이스가 있나? (있으면 청구 연동이 후속 과제로 커짐)
3. **세션별 인원** — 보강에 일부만 참석하는 걸 시간표/출결에서 구분 표시할 필요가 있나?
4. **휴강↔보강 강제 연결** — 휴강 시 반드시 보강을 잡아야 하나, 휴강 단독도 허용하나?
5. **주차 기준** — 한 주의 시작 요일(월요일 기준 가정). 토요일 중심 운영이라 다른 기준이 더 편한가?
